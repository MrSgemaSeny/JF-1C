Что я понял про продукт
JF-1C / Zhan Finance — это не personal finance и не бухгалтерия в классическом смысле. Это CRM-система для бухгалтерской/юридической фирмы с интеграцией в 1С. Три роли: ADMIN (управляет всем), EMPLOYEE (ведёт клиентов и задачи), CLIENT (видит свои документы и статус задач).
Реально работает: авторизация с JWT + refresh, CRUD задач с подзадачами и статусами, просмотр клиентов, загрузка/скачивание документов, базовый admin-дашборд.

Чего не хватает — честный разбор по приоритетам
🔴 Критично для выхода на рынок (без этого не продать)
1. Реальная интеграция с 1С

В названии есть "1C", в services.md наверняка описана — но в коде её нет. Это главное ценностное предложение. Сейчас это просто CRM. Без синхронизации с 1С ты конкурируешь с Bitrix24, amoCRM и сотней других — и проигрываешь. Нужно либо 1С-коннектор через COM/REST API 1С, либо импорт/экспорт в форматах, которые 1С понимает (XML, CSV по шаблону).
2. Уведомления

Клиент сдал документы — сотрудник не знает. Задача стала DONE — клиент не знает. Без уведомлений (email минимум, Telegram опционально) продукт работает только если все сидят и F5 жмут. Это блокер для продаж.
3. Нет тестов

В build.gradle есть тестовые зависимости (H2, spring-boot-starter-*-test), но реального покрытия скорее всего нет. Для B2B-продажи это важно: клиенты спрашивают про надёжность, и ты должен уметь отвечать.
4. Деплой

vite.config.ts говорит base: '/JF-1C/' — это GitHub Pages, не production. Нет docker-compose для backend + PostgreSQL. Показать продукт клиенту сейчас нельзя без локального запуска.

🟡 Важно для удержания (клиент ушёл, потому что этого нет)
5. Комментарии/чат внутри задачи

Сейчас TaskDetailsPage показывает задачу и подзадачи, но нет коммуникации. Клиент не может написать "уточните вот это", сотрудник не может попросить доп. документы прямо в задаче. Это первое, что спросят пользователи.
6. История изменений задачи (audit log)

Кто и когда перевёл задачу из IN_PROGRESS в ON_REVIEW? Для юридических/бухгалтерских процессов это критично. Сейчас этого нет.
7. Роли неполные

ADMIN регистрирует всех как CLIENT. Нет флоу приглашения сотрудника — очевидно придётся делать вручную через БД или добавить endpoint. Это больно при onboarding.
8. Subtask-статус не персистится

Посмотри на handleToggleSubtask — он меняет state локально, но не вызывает API. То есть изменения подзадач теряются при перезагрузке. Это баг, не фича.

🟢 Вектор развития (итерации после MVP)
Итерация 1 — "Показать клиенту" (2-3 недели)

Docker Compose (backend + postgres + nginx + frontend)
Починить subtask persistence
Email-уведомления при смене статуса задачи (Spring Mail + шаблоны)
Приглашение сотрудника по email (не только регистрация)

Итерация 2 — "Продать первому клиенту" (1-2 месяца)

1С-интеграция: минимум — импорт контрагентов из CSV/XML 1С
Комментарии в задаче
Audit log (кто/когда менял статус)
Базовые тесты на service-layer (хотя бы happy path)

Итерация 3 — "Масштабирование" (после первых 3-5 клиентов)

Multi-tenancy (сейчас один ADMIN на всё — значит один клиент на инстанс)
Платёжная интеграция (если хочешь SaaS)
Мобильная адаптация (сейчас h-screen flex — на телефоне неудобно)
Real-time обновления (WebSocket или SSE вместо polling)


Главный подводный камень, который ты можешь не видеть
Нет multi-tenancy. Сейчас архитектура предполагает одну компанию на весь инстанс. Если ты хочешь B2B SaaS (продавать нескольким фирмам), нужен Organization entity и tenant isolation — это архитектурное изменение, которое болезненно делать потом. Реши сейчас: one-tenant per deploy (проще, можно продавать как "своя установка") или multi-tenant SaaS (сложнее, но масштабируется).
Для старта рекомендую one-tenant per deploy — это честнее, быстрее и понятнее для бухгалтерских фирм, которые параноят насчёт данных.
