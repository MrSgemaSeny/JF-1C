# План: личные кабинеты по ролям (Admin / Employee / Client) + CRM-модуль

Статус: в разработке.
Связанные документы: `ARCHITECTURE.md` (общая архитектура), `simple-fsd-1c.md`, `services.md`.

## 0. Контекст и что уже есть в репозитории

Прежде чем писать что-то новое, зафиксируем то, что уже реализовано — чтобы не дублировать:

**Backend** (`zhan-finance-backend`, Spring Boot, модульный монолит по доменам):
- `modules/auth` — JWT access/refresh, login/register, `Role { ADMIN, EMPLOYEE, CLIENT }`, `User.assignedEmployee` (клиент уже привязывается к сотруднику).
- `modules/billing` — Invoice/Subscription, есть `*AccessService` с row-level проверками (admin видит всё, employee — только своих клиентов, client — только себя). Это эталонный паттерн, по нему делаем CRM.
- `modules/user` — `/api/users/me` профиль.
- `common/response/ApiResponse<T>` — единый конверт ответа `{success, data, message, timestamp}`.
- `common/exception` — `ApiException` + `ErrorCode` + `GlobalExceptionHandler`.

**Frontend** (`zhan-finance-frontend`, FSD, React 19 + Vite + Tailwind v4):
- `features/auth/AuthContext.tsx` — уже хранит `role` в `StoredAuth`, обновляет токен на 401, контекст полностью готов к ролевой логике.
- `shared/api/http.ts` — типизированный `apiRequest<T>`, разворачивает `ApiResponse`, сам рефрешит токен.
- `pages/profile/ProfilePage.tsx` — текущая заглушка ЛК без разделения по ролям. **Заменяем на ролевой роутинг**, как договорились.
- Фирменный стиль: `brand-green` (#00562D) / `brand-beige` (#F5F5DC), компоненты `Button`, `Section`, `Container`.

**Чего не хватает:** CRM-модуля (clients, tasks) — есть только в виде SQL-наброска в `ARCHITECTURE.md`, в коде не реализован.

## 1. Цель

1. Развести `/profile` на три полноценных раздела по роли пользователя:
   - `/admin/*` — сотрудники, клиенты, задачи, метрики компании;
   - `/employee/*` — CRM рабочее место: свои клиенты, свои задачи, отчёты;
   - `/client/*` — свои задачи/заявки и их статус, свои документы/счета.
2. Добавить backend CRM-модуль (`clients`, `tasks`) с тем же паттерном row-level security, что и в `billing`.
3. Сделать фронт так, чтобы переход с mock-данных на реальный backend требовал **только переключения одного флага/слоя данных**, без переписывания компонентов.

## 2. Backend: CRM-модуль

### 2.1 Структура пакета
```
modules/crm/
├── controller/
│   ├── ClientController.java      → /api/crm/clients
│   └── TaskController.java        → /api/crm/tasks
├── dto/
│   ├── ClientDto.java
│   ├── ClientCreateRequest.java
│   ├── TaskDto.java
│   ├── TaskCreateRequest.java
│   └── TaskStatusUpdateRequest.java
├── entity/
│   ├── ClientProfile.java   (расширение карточки клиента сверх User: компания, заметки)
│   └── Task.java
├── repository/
│   ├── ClientProfileRepository.java
│   └── TaskRepository.java
└── service/
    ├── ClientService.java
    ├── TaskService.java
    └── CrmAccessService.java   (по образцу InvoiceAccessService)
```

Примечание: «клиент» как пользователь уже есть (`User` с `role=CLIENT`, `assignedEmployee`). `ClientProfile` — это 1:1 доп.карточка (компания, телефон, заметки CRM), а не дублирование User.

### 2.2 Модель данных (новая миграция `V3__Crm_Schema.sql`)
```sql
CREATE TABLE client_profiles (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY,
    user_id BIGINT NOT NULL UNIQUE REFERENCES app_users(id) ON DELETE CASCADE,
    company_name VARCHAR(255),
    phone VARCHAR(32),
    notes TEXT,
    created_at TIMESTAMP(6) WITH TIME ZONE NOT NULL,
    updated_at TIMESTAMP(6) WITH TIME ZONE NOT NULL,
    PRIMARY KEY (id)
);

CREATE TABLE tasks (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY,
    title VARCHAR(255) NOT NULL,
    description TEXT,
    client_id BIGINT NOT NULL REFERENCES app_users(id) ON DELETE CASCADE,
    assigned_to_id BIGINT REFERENCES app_users(id) ON DELETE SET NULL,
    status VARCHAR(32) NOT NULL DEFAULT 'NEW' CHECK (status IN ('NEW','IN_PROGRESS','ON_REVIEW','DONE','CANCELLED')),
    priority VARCHAR(16) NOT NULL DEFAULT 'MEDIUM' CHECK (priority IN ('LOW','MEDIUM','HIGH','URGENT')),
    due_date DATE,
    created_by_id BIGINT NOT NULL REFERENCES app_users(id),
    created_at TIMESTAMP(6) WITH TIME ZONE NOT NULL,
    updated_at TIMESTAMP(6) WITH TIME ZONE NOT NULL,
    PRIMARY KEY (id)
);

CREATE INDEX idx_tasks_client ON tasks(client_id);
CREATE INDEX idx_tasks_assigned_to ON tasks(assigned_to_id);
CREATE INDEX idx_tasks_status ON tasks(status);
CREATE INDEX idx_client_profiles_user ON client_profiles(user_id);
```

### 2.3 Доступы (row-level), по аналогии с `InvoiceAccessService`
| Действие | ADMIN | EMPLOYEE | CLIENT |
|---|---|---|---|
| Список клиентов | все | только свои (`assignedEmployee == me`) | — |
| Список задач | все | только задачи своих клиентов / назначенные себе | только свои задачи (`task.client == me`) |
| Создать задачу | да | да (только для своих клиентов) | нет (может только подать заявку — отдельный эндпоинт `POST /api/crm/tasks/request`, создаётся в статусе `NEW` без `assignedTo`) |
| Сменить статус задачи | да | да (если назначена ему) | нет (только просмотр статуса) |
| Назначить сотрудника на клиента | да | нет | — |

### 2.4 Эндпоинты
```
GET    /api/crm/clients                    ADMIN, EMPLOYEE
GET    /api/crm/clients/{id}                ADMIN, EMPLOYEE (row-level)
PATCH  /api/crm/clients/{id}                ADMIN, EMPLOYEE
POST   /api/crm/clients/{id}/assign         ADMIN — назначить employee

GET    /api/crm/tasks                       ADMIN, EMPLOYEE, CLIENT (row-level фильтр внутри сервиса)
GET    /api/crm/tasks/{id}                  row-level
POST   /api/crm/tasks                       ADMIN, EMPLOYEE
POST   /api/crm/tasks/request               CLIENT — заявка на задачу
PATCH  /api/crm/tasks/{id}/status           ADMIN, EMPLOYEE
PATCH  /api/crm/tasks/{id}/assign           ADMIN, EMPLOYEE — назначить/переназначить исполнителя

GET    /api/crm/dashboard/admin             ADMIN — агрегаты (кол-во клиентов/сотрудников/задач по статусам)
GET    /api/crm/dashboard/employee          EMPLOYEE — агрегаты по своим клиентам/задачам
GET    /api/crm/dashboard/client            CLIENT — агрегаты по своим задачам

GET    /api/admin/users                     ADMIN — список всех сотрудников/клиентов (для админ-страницы)
PATCH  /api/admin/users/{id}/role           ADMIN — смена роли (опционально, можно отложить)
```

`SecurityConfig` расширяется через `@PreAuthorize` на контроллерах (как уже сделано в `InvoiceController`), глобальный `authorizeHttpRequests` менять не нужно.

### 2.5 Реализация — порядок
1. Миграция `V3__Crm_Schema.sql`.
2. Entities: `ClientProfile`, `Task` (+ enum `TaskStatus`, `TaskPriority`).
3. Repositories с `@Query` для join-fetch (по примеру `InvoiceRepository.findAllWithClient`).
4. `CrmAccessService` — read/write/create проверки.
5. `ClientService`, `TaskService` — бизнес-логика + маппинг в DTO.
6. Контроллеры с `@PreAuthorize`.
7. Dashboard-агрегаты — простые `COUNT`/`GROUP BY` через repository-методы, не отдельный модуль аналитики (overkill для MVP).
8. Тест: расширить `ApiSmokeTests` сценарием создания задачи и проверки 403 у чужого сотрудника.

## 3. Frontend: архитектура личных кабинетов

### 3.1 Маршруты
```
/admin                 → AdminDashboardPage (метрики компании)
/admin/employees        → список сотрудников
/admin/clients          → список клиентов
/admin/tasks             → все задачи (фильтры)

/employee               → EmployeeDashboardPage (мои метрики)
/employee/clients        → мои клиенты
/employee/tasks           → мои задачи

/client                 → ClientDashboardPage
/client/tasks             → мои заявки/задачи + статус
/client/documents          → мои документы/счета (использует уже существующий billing API)

/profile  → редиректит на /admin, /employee или /client по user.role (тонкий компонент-роутер)
```

`ROUTES` в `shared/config/routes.ts` расширяется этими путями. `/profile` остаётся как «вход в кабинет» — единая ссылка в шапке сайта, не зависящая от роли.

### 3.2 Ролевой роутинг — `RoleProtectedRoute`
Текущий `ProtectedRoute` просто проверяет `user != null`. Добавляем поверх него `RoleGate`:

```tsx
// features/auth/RoleProtectedRoute.tsx
export function RoleProtectedRoute({ allow }: { allow: UserRole[] }) {
  const { user } = useAuth();
  if (!user) return <Navigate to={ROUTES.LOGIN} replace />;
  if (!allow.includes(user.role)) return <Navigate to={roleHome(user.role)} replace />;
  return <Outlet />;
}
```
`roleHome(role)` — единая функция «куда вести пользователя его роли» (`/admin` | `/employee` | `/client`), используется и в `/profile`-редиректе, и после логина.

### 3.4 Слой данных, готовый к подключению backend без переписывания компонентов

Ключевой принцип: **компоненты страниц никогда не дёргают mock или http напрямую** — только через `entities/*/api/*Api.ts`, который имеет один и тот же контракт в обоих режимах.

```typescript
// entities/task/api/taskApi.ts
import { apiRequest } from '@/shared/api/http';
import { USE_MOCK_API } from '@/shared/config/env';
import * as mock from './taskApi.mock';
import type { TaskDto, TaskStatus } from '../model/types';

export async function getTasks(filter?: TaskFilter): Promise<TaskDto[]> {
  if (USE_MOCK_API) return mock.getTasks(filter);
  return apiRequest<TaskDto[]>(`/api/crm/tasks${toQuery(filter)}`);
}

export async function updateTaskStatus(id: number, status: TaskStatus): Promise<TaskDto> {
  if (USE_MOCK_API) return mock.updateTaskStatus(id, status);
  return apiRequest<TaskDto>(`/api/crm/tasks/${id}/status`, {
    method: 'PATCH',
    body: JSON.stringify({ status }),
  });
}
```

- `USE_MOCK_API` — один флаг в `shared/config/env.ts`, читает `import.meta.env.VITE_USE_MOCK_API` (по умолчанию `true`, пока бэкенд не задеплоен туда же, куда фронт на GitHub Pages).
- `taskApi.mock.ts` лежит рядом, реализует тот же сигнатурный контракт, использует in-memory массив + `localStorage` (не браузерный storage артефактов — обычный сайт, ограничение про артефакты сюда не относится) для персистентности между перезагрузками демо.
- DTO-типы (`model/types.ts`) пишутся вручную **зеркально backend DTO** (`TaskDto`, `ClientDto` и т.д.), с комментарием `// зеркалит modules/crm/dto/TaskDto.java`. Когда подключаем OpenAPI-генерацию (см. `ARCHITECTURE.md` §3.2), эти файлы заменяются на сгенерированные — остальной код не меняется, потому что импортирует типы по тем же именам из того же `@/entities/task` барреля.
- Когда backend готов: меняем `VITE_USE_MOCK_API=false` в `.env`, всё работает на реальном API — **ноль изменений в `pages/`, `widgets/`, `features/`**.

### 3.5 Состав «отлично» — что входит в каждый кабинет

**Admin (`/admin`)**
- Overview: карточки-метрики (всего клиентов, сотрудников, задач по статусам, просроченные), графика не делаем (overkill для MVP, просто цифры + список последних задач).
- Employees: таблица сотрудников (имя, email, кол-во клиентов, кол-во активных задач), поиск.
- Clients: таблица клиентов с назначенным сотрудником, возможность переназначить (`client-assign` feature), поиск/фильтр.
- Tasks: все задачи компании, фильтр по статусу/сотруднику/клиенту, смена статуса.

**Employee (`/employee`)**
- Overview: мои метрики (мои клиенты, мои активные задачи, дедлайны на этой неделе).
- Clients: список «моих» клиентов, карточка клиента с историей задач.
- Tasks: канбан/список моих задач, смена статуса, создание новой задачи для своего клиента.

**Client (`/client`)**
- Overview: статус последних заявок, краткая сводка.
- Tasks: список своих задач/заявок со статусами (read-only + кнопка «оставить заявку» → `POST /tasks/request`).
- Documents: переиспользует уже существующий `billing` (Invoice) API — счета и документы клиента.

### 3.6 UI / дизайн-система кабинетов
- Общий `DashboardLayout`: левый сайдбар (лого, пункты меню по роли, профиль внизу с logout), брендовые цвета `brand-green`/`brand-beige`, та же типографика, что на публичном сайте — кабинет не должен выглядеть как сторонний продукт.
- Переиспользуем `shared/ui/Button`, добавляем по необходимости `shared/ui/Badge` (статусы задач), `shared/ui/Table`, `shared/ui/StatCard` — как новые примитивы в том же `shared/ui`.
- Статусы задач — цветные бейджи (`NEW` серый, `IN_PROGRESS` жёлтый, `ON_REVIEW` синий, `DONE` зелёный, `CANCELLED` красный/приглушённый).

## 4. Этапы реализации (порядок коммитов)

1. **Backend CRM-модуль**: миграция → entities → repositories → access service → services → controllers → smoke-тест.
2. **Frontend data layer**: `entities/{client,task,employee}` с типами + mock-реализацией + флагом `USE_MOCK_API`.
3. **Frontend shell**: `DashboardLayout`, `DashboardSidebar`, `nav-config.ts`, `RoleProtectedRoute`, `roleHome()`.
4. **Frontend pages**: admin → employee → client, по одной странице за раз, на mock-данных.
5. **Роутинг**: обновить `App.tsx` и `routes.ts`, заменить `/profile` на `DashboardRedirect`.
6. **Связка с реальным backend**: выставить `VITE_USE_MOCK_API=false` локально, прогнать вручную против поднятого backend, поправить расхождения DTO, если найдутся.
7. (Опционально, после MVP) Подключить `openapi-typescript` генерацию типов по `ARCHITECTURE.md §3.2`, заменить ручные DTO на сгенерированные.

## 5. Что сознательно остаётся вне MVP
- Канбан с drag-n-drop (делаем список с фильтром по статусу + быстрая смена статуса через select/кнопки — drag-n-drop можно добавить позже без изменения API).
- Графики/аналитика на overview — только цифры.
- Внутренняя переписка/уведомления.
- Интеграция с 1С (по `ARCHITECTURE.md` остаётся заглушкой).
- Смена роли пользователя через UI (есть только в плане эндпоинт, без экрана).
