## 2. Критически не хватает (без Docker/деплоя)

### 2.1 Безопасность — приоритет P0
1. **`/uploads/**` открыт `permitAll()`** в `SecurityConfig.java` — любой, зная
   путь к файлу, скачивает документы клиента без авторизации. Нужен отдельный
   защищённый контроллер выдачи файлов с проверкой владения/роли, а не статик-серв.
2. **Swagger/OpenAPI публичны** в проде (`/swagger-ui/**`, `/v3/api-docs/**`).
   Минимум — спрятать за профиль `dev`/`local`, в проде выключать.
3. **Нет rate limiting** на `/api/auth/**` (login, register, refresh) — открыт
   брутфорс и enumeration атаки. Добавить Bucket4j или Spring Cloud Gateway
   на уровне фильтра.
4. **`ddl-auto=update` + Flyway одновременно** — конфликт источников истины
   схемы. Переключить на `validate` в проде, схему вести только миграциями.
5. **Дефолтный JWT secret** (`change-me-change-me...`) в `application.properties`
   как fallback — если переменная окружения не задана, сервис стартует с
   предсказуемым секретом. Сделать обязательным (fail-fast при отсутствии).
6. Нет проверки, что WebSocket STOMP-подключение к чату проверяет права
   пользователя на конкретную комнату/диалог (просмотреть `ChatController`/`WebSocketConfig`).

### 2.2 Бизнес-логика — приоритет P0/P1
1. **Нет реальной оплаты.** Invoice/Subscription — просто записи в БД со статусом.
   Для системы, позиционируемой как биллинг/бухгалтерия, это критично:
   нужен хотя бы один платёжный провайдер (Kaspi Pay/Stripe/эквайринг банка)
   и вебхук подтверждения оплаты.
2. **Нет полноценного audit log** — кто и когда менял статус задачи, сумму
   счёта, роль пользователя. `BaseEntity` даёт только createdAt/updatedAt,
   этого недостаточно для финансовой системы (нужна история изменений/кто изменил).
3. **Хранилище документов не production-ready**: локальный диск или БД —
   не переживёт редеплой/масштабирование без persistent volume или
   объектного хранилища. (Ты сам зафиксировал это в `documents.md`, но
   реализация ещё не выбрана.)
4. Модуль услуг (описанный в сессии) нужно либо восстановить, либо
   реализовать заново — это прямой источник дохода/лидов с сайта.

### 2.3 Тестирование — приоритет P1
- Backend: 6 тестовых файлов на весь монолит (auth unit test, task integration
  test, пара smoke-тестов). Нет тестов для billing, documents, chat, notifications, courses.
- Frontend: **0 тестов**. В `package.json` нет vitest/jest/testing-library вообще.
- Нет тестов авторизации по ролям (кто и что может делать — самое важное
  для RBAC-системы) кроме smoke-тестов.

### 2.4 Качество кода — приоритет P2
- Три параллельных компонента для деталей задачи:
  `entities/task/ui/TaskDetailsModal.tsx` (524 строки),
  `widgets/task-detail-modal/Taskdetailmodal.tsx` (25 строк, похоже на забытую заглушку),
  `features/task-management/ui/TaskDetailCard.tsx` (262 строки).
  Нужно выбрать один источник правды и удалить остальные.
- Redis в зависимостях `build.gradle`, но исключён в `application.properties` —
  мёртвая зависимость, либо подключить кэш реально, либо убрать.
- Нет глобального state/data-fetching слоя (TanStack Query/Zustand) —
  ручные fetch по компонентам, будет плодить дублирующиеся запросы и
  рассинхрон кэша по мере роста фич.
- `src_b.zip`, `src_f.zip`, `logs.txt`, `fly_logs.txt`, `prod_logs.txt`,
  `test-login.json`, `test-register.json` лежат в репозитории — это мусор/
  потенциальная утечка данных (логи, тестовые учётки), нужно в `.gitignore`
  и вычистить историю при необходимости.

### 2.5 Отсутствующие полноценные функции

## 3. План работ по приоритету

### P0 — сделать в первую очередь (безопасность и данные не должны утечь)
1. Закрыть `/uploads/**`, отдавать файлы через защищённый контроллер с проверкой владения.
2. Убрать Swagger/OpenAPI из прод-профиля.
3. Добавить rate limiting на auth endpoints.
4. `ddl-auto=validate` в проде, весь DDL только через Flyway.
5. Fail-fast, если `JWT_SECRET` не задан явно (убрать дефолт).
6. Проверить/добавить авторизацию на WebSocket комнаты чата.
7. Разобраться с пропавшим модулем услуг — найти в локальной копии или переделать.

### P1 — бизнес-критично
1. Интеграция платёжного провайдера для Invoice (хотя бы один способ оплаты + вебхук статуса).
2. Audit log для Task/Invoice/User (кто/когда/что изменил).
3. Выбрать постоянное хранилище документов (S3-совместимое, например Cloudflare R2/MinIO) вместо БД/локального диска.
4. Базовый набор backend-тестов на billing и documents (deньги и файлы — самое рискованное).
5. Восстановить/реализовать модуль услуг с привязкой к задачам (как было описано).

### P2 — качество и устойчивость
1. Убрать дубликаты task-detail компонентов, оставить один.
2. Добавить vitest + React Testing Library, покрыть критичные формы (логин, создание задачи, создание счёта).
3. Ввести TanStack Query для данных с сервера — избавиться от ручных fetch-паттернов.
4. Убрать мусорные файлы из репозитория (`src_b.zip`, `logs.txt`, `test-login.json` и т.д.), добавить в `.gitignore`.
5. Решить судьбу Redis-зависимости (подключить кэш или убрать пакет).

### P3 — расширение функциональности
1. Email-верификация регистрации, при отсутствии — добавить.
2. 2FA для ADMIN.
3. Экспорт/импорт в формат 1С.
4. i18n-слой (даже если сейчас только RU, заложить структуру).
5. Soft-delete + экспорт персональных данных клиента.
