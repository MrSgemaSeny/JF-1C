## Шаг 1.1 — Модель данных: Сделки и Воронка

*Прежде чем рисовать канбан — нужно решить, как устроены данные. Это самое важное решение во всём плане.*

### Что нужно понять до начала

Воронка (Pipeline) и Стадия (Stage) — разные сущности. Воронок может быть несколько (продажи, партнёры, рекрутинг). У каждой воронки — свои стадии. Сделка принадлежит конкретной воронке и находится в конкретной стадии.

```
Pipeline (Воронка)
  └── Stage[] (Стадии, упорядоченные)
        └── Deal[] (Сделки в этой стадии)
```

### Сущности и поля

**Pipeline (Воронка)**
- `id` — UUID
- `name` — название ("Общая воронка", "Партнёры")
- `isDefault` — одна воронка по умолчанию
- `createdAt`, `updatedAt`

**Stage (Стадия)**
- `id` — UUID
- `pipelineId` — к какой воронке
- `name` — название ("Новая", "Переговоры", "КП отправлено")
- `order` — порядок (1, 2, 3...) — именно через это поле, не через порядок в массиве
- `color` — цвет колонки (hex)
- `type` — enum: `OPEN`, `WON`, `LOST` — важно для аналитики
- `isDefault` — стадия по умолчанию для новых сделок

**Deal (Сделка)**
- `id` — UUID
- `title` — название сделки
- `amount` — сумма (decimal, не float — деньги нельзя хранить в float)
- `currency` — KZT / USD / RUB (enum)
- `stageId` — текущая стадия
- `pipelineId` — воронка
- `contactId` — контакт (nullable — может не быть)
- `companyId` — компания (nullable)
- `responsibleUserId` — ответственный
- `source` — источник лида (enum: INSTAGRAM, WEBSITE, REFERRAL, COLD_CALL, OTHER)
- `closedAt` — дата ожидаемого закрытия
- `lostReason` — причина провала (nullable, заполняется при переводе в LOST)
- `tags` — массив строк или связь с таблицей тегов
- `createdAt`, `updatedAt`, `deletedAt` — **soft delete обязателен**

### Почему именно так

`order` как отдельное поле позволяет перетаскивать стадии без перезаписи всех записей. `type: WON/LOST` позволяет считать конверсию без хардкода названий стадий. `amount` в decimal — потому что `0.1 + 0.2 = 0.30000000000000004` в float, и это реальная проблема в финансах.

## Шаг 1.2 — API для сделок

*Минимальный набор эндпоинтов, который нужен канбану.*

### Эндпоинты

```
GET    /api/pipelines                        — список всех воронок
GET    /api/pipelines/{id}/stages            — стадии воронки с количеством и суммой сделок
GET    /api/pipelines/{id}/deals             — все сделки воронки (с пагинацией и фильтром)
POST   /api/deals                            — создать сделку
GET    /api/deals/{id}                       — карточка сделки
PUT    /api/deals/{id}                       — обновить поля сделки
PATCH  /api/deals/{id}/stage                 — сменить стадию (отдельный эндпоинт — важно)
DELETE /api/deals/{id}                       — soft delete (deletedAt = now)
```

### Почему `PATCH /stage` отдельно

Смена стадии — это не просто обновление поля. Это событие: нужно записать в историю, кто и когда перевёл сделку. Если делать через общий `PUT /deals/{id}` — теряешь этот контекст. Отдельный эндпоинт = отдельная логика = чистая история.

### Что возвращает `GET /pipelines/{id}/stages`

```json
[
  {
    "id": "uuid",
    "name": "Новая",
    "order": 1,
    "color": "#6366f1",
    "type": "OPEN",
    "dealsCount": 3,
    "totalAmount": 1500000,
    "currency": "KZT"
  }
]
```

Агрегация `dealsCount` и `totalAmount` считается на бэке, не на фронте. Не тащи все сделки на фронт, чтобы посчитать сумму — это медленно и неправильно.

## Шаг 1.3 — Канбан-доска: фронтенд

*Визуально самая впечатляющая часть. Делай её последней из этого шага — после того как API работает.*

### Структура компонентов

```
KanbanPage
├── PipelineSelector          — переключатель воронок
├── KanbanBoard
│   ├── KanbanColumn[]        — одна колонка = одна стадия
│   │   ├── ColumnHeader      — название, счётчик, сумма
│   │   ├── DealCard[]        — карточки сделок
│   │   │   ├── DealTitle
│   │   │   ├── DealAmount
│   │   │   ├── ContactName
│   │   │   ├── DaysInStage   — сколько дней в текущей стадии
│   │   │   └── QuickActions  — звонок, email, добавить дело
│   │   └── AddDealButton     — быстрое создание сделки в этой стадии
│   └── LostColumn            — отдельная, визуально другая
└── CreateDealModal
```

### Drag & Drop

Используй `@dnd-kit/core` — это современная замена react-beautiful-dnd (он устарел и не поддерживается).

Логика при drop:
1. Optimistic update — сразу двигаем карточку визуально
2. `PATCH /deals/{id}/stage` — отправляем запрос
3. Если ошибка — откатываем обратно (TanStack Query это умеет)

Не жди ответа сервера, чтобы двигать карточку — пользователь почувствует лаг.

### На что обратить внимание

Когда пользователь перетаскивает сделку в стадию с `type: LOST` — показывай модалку с вопросом "Почему сделка провалена?". Не давай провалить сделку без причины — эти данные потом нужны для аналитики.

## Шаг 1.4 — Dashboard

*Делается после канбана, потому что большинство данных для дашборда ты уже умеешь получать.*

### Что строим

Dashboard — это не набор красивых карточек. Это ответ на вопрос "что мне делать прямо сейчас". Каждый блок должен либо показывать проблему, либо давать повод для действия.

### API для Dashboard

Один эндпоинт, не пять:

```
GET /api/dashboard/summary
```

Возвращает:

```json
{
  "todayActivities": {
    "count": 4,
    "items": [...]
  },
  "overdueActivities": {
    "count": 2,
    "items": [...]
  },
  "activeDeals": {
    "count": 12,
    "totalAmount": 4500000,
    "currency": "KZT"
  },
  "closedThisMonth": {
    "count": 3,
    "totalAmount": 900000
  },
  "recentActivity": [...]
}
```

Всё в одном запросе — потому что дашборд должен грузиться быстро. Не делай 5 параллельных запросов для 5 блоков.

### Структура компонентов

```
DashboardPage
├── StatsRow
│   ├── StatCard "Дела сегодня"       — число + клик ведёт в список
│   ├── StatCard "Просрочено"          — красный, если > 0
│   ├── StatCard "Сделок в работе"     — число и сумма
│   └── StatCard "Закрыто за месяц"    — число и сумма
├── TodayActivities                    — список дел на сегодня
│   └── ActivityItem[]
│       ├── ActivityType icon
│       ├── Description
│       ├── LinkedDeal / Contact
│       └── Actions: Выполнено / Перенести
└── RecentActivity                     — лента последних изменений
    └── ActivityFeedItem[]             — "Иван перевёл сделку 'Алибек' в стадию В работе"
```

### Пустые состояния — не забудь

Когда дел нет — не показывай пустую карточку. Напиши что-то конкретное: "На сегодня дел нет — отличный повод создать сделку". Пустой экран демотивирует.

### Частота обновления

Dashboard не нужен в реальном времени. Кэшируй через TanStack Query с `staleTime: 60 * 1000` (1 минута). Добавь кнопку "Обновить" — пользователь сам решит когда обновить.

## Шаг 2.1 — Контакты: модель данных

*Контакт — центральная сущность CRM. Вокруг него строится всё.*

### Сущности

**Company (Компания)**
- `id` — UUID
- `name` — название
- `inn` — ИНН (строка, не число — ИНН может начинаться с нуля)
- `industry` — сфера деятельности
- `website` — сайт
- `address` — адрес
- `phone` — основной телефон
- `createdAt`, `updatedAt`, `deletedAt`

**Contact (Контакт)**
- `id` — UUID
- `firstName`, `lastName` — раздельно (чтобы сортировать по фамилии)
- `companyId` — компания (nullable)
- `position` — должность
- `phone` — нормализованный формат `+7XXXXXXXXXX`
- `email`
- `whatsapp` — номер WhatsApp (часто отличается от основного)
- `telegram` — username или номер
- `source` — enum: INSTAGRAM, WEBSITE, REFERRAL, COLD_CALL, EXHIBITION, OTHER
- `tags` — теги
- `notes` — заметки (свободный текст)
- `createdAt`, `updatedAt`, `deletedAt`

### Про телефоны

Все телефоны хранить в нормализованном виде: `+77073495503`. При сохранении — нормализовать на бэке, не на фронте. Пользователь вводит как угодно, система хранит единообразно.

Напиши один метод нормализации и используй его везде:
- `8 707 349 5503` → `+77073495503`
- `87073495503` → `+77073495503`
- `+7 (707) 349-55-03` → `+77073495503`

## Шаг 2.4 — Активности (Дела)

*Это связующая ткань CRM. Без активностей — нет истории, нет контекста.*

### Модель данных

**Activity**
- `id` — UUID
- `type` — enum: CALL, EMAIL, MEETING, TASK, MESSAGE
- `title` — краткое описание ("Позвонить по итогам КП")
- `description` — детали (опционально)
- `dueDate` — дедлайн (date + time)
- `completedAt` — когда выполнено (null = не выполнено)
- `status` — computed: PLANNED / OVERDUE / COMPLETED
- `dealId` — привязка к сделке (nullable)
- `contactId` — привязка к контакту (nullable)
- `responsibleUserId` — ответственный
- `createdByUserId` — кто создал
- `createdAt`, `updatedAt`

Правило: у активности должна быть хотя бы одна привязка — либо к сделке, либо к контакту (или обе).

### API

```
GET    /api/activities                        — мои активности (фильтр: today, overdue, all)
POST   /api/activities                        — создать
GET    /api/activities/{id}                   — детали
PATCH  /api/activities/{id}/complete          — отметить выполненной
PUT    /api/activities/{id}                   — редактировать
DELETE /api/activities/{id}                   — удалить
GET    /api/deals/{id}/activities             — активности сделки
GET    /api/contacts/{id}/activities          — активности контакта
```

### Фронтенд: форма создания активности

Форма появляется в трёх местах: на карточке сделки, на карточке контакта, на дашборде.

Делай её переиспользуемым компонентом с пропсами `dealId?` и `contactId?`.

```

├── TypeSelector          — иконки типов (CALL / EMAIL / MEETING / TASK / MESSAGE)
├── TitleInput
├── DateTimePicker        — дедлайн
├── ResponsibleSelect     — кто отвечает
├── DescriptionTextarea   — опционально
└── Submit
```

### Таймлайн на карточке сделки

Таймлайн — хронологический список всего, что происходило со сделкой:

```
TimelineItem
├── [иконка типа]  [дата]  [ответственный]
├── Текст события
└── Actions (если активность — Выполнено / Редактировать / Удалить)
```

Типы событий в таймлайне:
- Активность (создана / выполнена)
- Смена стадии
- Изменение суммы
- Комментарий
- Создание сделки

Запрос: `GET /api/deals/{id}/timeline` — возвращает всё в хронологическом порядке, смешанно.

## Шаг 2.5 — Теги

*Маленькая фича, которую легко сделать правильно с первого раза.*

### Два подхода — выбери один

**Вариант A: Теги как строки в массиве**
Просто. Теги хранятся прямо в сущности (Deal, Contact). Минус: нет консистентности — один написал "VIP", другой "vip", третий "випп".

**Вариант B: Теги как отдельная таблица**
Таблица `tags` + таблица `deal_tags` / `contact_tags`. Теги создаются один раз, переиспользуются. При вводе — автокомплит по существующим тегам.

Рекомендация: **Вариант B**. Немного сложнее, но данные чистые и можно потом фильтровать по тегу.

### API

```
GET    /api/tags                     — все теги (для автокомплита)
POST   /api/tags                     — создать тег
POST   /api/deals/{id}/tags          — добавить тег к сделке
DELETE /api/deals/{id}/tags/{tagId}  — убрать тег
POST   /api/contacts/{id}/tags       — добавить тег к контакту
```

### Фронтенд

Компонент `TagInput` — поле с автокомплитом. Вводишь "VI" — видишь "VIP". Нет такого тега — предлагает создать. Выбранные теги показываются как chips с крестиком.

Переиспользуй этот компонент везде, где нужны теги.

# Общие принципы для всего плана

## Порядок работы над каждой фичей

Каждую фичу делай в таком порядке, не меняй его:

```
1. Миграция БД (добавить таблицы / поля)
2. Модель (Entity / DTO)
3. Репозиторий (запросы к БД)
4. Сервис (бизнес-логика)
5. Контроллер (эндпоинты)
6. Тест на сервис (минимум happy path)
7. Фронт: TanStack Query хук
8. Фронт: компоненты
9. Ручное тестирование в браузере
```

Не начинай фронт, пока API не работает. Не начинай API, пока нет миграции. Порядок важен.

## Как называть вещи (единый язык)

Договорись с собой один раз и не меняй:

| Понятие | В коде (EN) | В UI (RU) |
|---|---|---|
| Сделка | Deal | Сделка |
| Воронка | Pipeline | Воронка |
| Стадия | Stage | Стадия |
| Дело / активность | Activity | Дело |
| Контакт | Contact | Контакт |
| Компания | Company | Компания |
| Ответственный | ResponsibleUser | Ответственный |
| Источник | Source | Источник |

Если в коде `Deal`, а в UI "Задача" — это путаница. Единый язык экономит время.

## Три вещи, которые нельзя откладывать

**Soft delete с первого дня.** Поле `deletedAt` добавляй в каждую новую сущность сразу. Не потом. Один раз забудешь — и через месяц будешь переписывать миграции и весь слой данных.

**UUID вместо auto-increment ID.** Если ещё не перешёл — переходи сейчас. UUID позволяет создавать сущности на клиенте до ответа сервера (нужно для optimistic update на канбане).

**Валидация телефонов на бэке.** Напиши утилиту нормализации телефона один раз. Используй её везде где принимаешь телефон на вход. Потом когда добавишь WhatsApp-интеграцию — скажешь себе спасибо.
